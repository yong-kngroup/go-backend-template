# 工程约束

本文档定义 `go-backend-template` 作为可复用 Go 后端模板的维护规则。目标是让模板主线始终可运行、可验证、可部署，并让具体业务项目能够持续吸收已验证的通用改进。

代码入口和关键运行链路见 [架构导航](architecture.md)。

## 1. 分支与版本策略

### 1.1 分支职责

- `main`：模板主线。只包含通用能力、示例配置、基础设施适配和与业务无关的示例实现。
- `project/<name>`：从模板版本切出的具体业务项目分支。可包含业务领域、业务表、业务接口和业务部署配置。
- `feature/<name>`：短生命周期功能分支，合并后删除。
- `fix/<name>`：缺陷修复分支；应优先修复对应的主线或业务分支。

`main` 不得出现具体客户、商品、订单、业务字段、业务流程或仅服务于单个项目的配置。

### 1.2 创建业务项目

新业务项目必须从已发布的模板 tag 创建，而不是从未标记的任意提交创建：

```text
template/v0.1.0
  -> project/example-service
```

每个模板 tag 必须能独立构建、通过质量门槛，并有对应的变更说明。

### 1.3 回灌模板的规则

在业务项目中发现的改进，按以下原则处理：

- 与业务无关、可复用、已在真实场景验证的改进，应提炼为独立提交并合入 `main`。
- 业务模型、业务规则和业务集成只保留在 `project/<name>`。
- 回灌前必须删除业务命名、私有地址、业务数据和特定流程假设。
- 不直接将整个业务分支合并回 `main`；应通过独立 PR 或 `cherry-pick` 回灌通用提交。

## 2. 架构边界

项目遵循以下依赖方向：

```text
handler -> usecase -> domain
                  -> repository / infra
cmd -> handler / usecase / infra 的装配
```

- `cmd/server` 仅负责 HTTP 服务启动和依赖装配。
- `cmd/worker` 仅负责消费异步消息和依赖装配。
- `cmd/cron` 仅负责调度定时任务和依赖装配。
- `handler` 仅负责协议转换、输入校验、认证授权和 HTTP 响应。
- `usecase` 负责业务流程编排、事务边界和调用领域能力。
- `domain` 定义实体、领域规则、事件和依赖契约；不得依赖 Gin、GORM、Redis、Kafka 等具体实现。
- `repository` 负责持久化实现；不得承载业务流程。
- `infra` 负责数据库、缓存、消息、日志、追踪、加密等外部技术适配。
- `pkg` 只放可脱离本项目业务语义复用的技术组件。

禁止在 Handler 中直接操作 GORM、Redis 或 Kafka；禁止在 Usecase 中直接依赖具体 Kafka、Redis 或 GORM 类型。

### 2.1 HTTP 响应约定

- 面向前端的业务 API，无论成功还是失败，HTTP 状态码统一返回 `200`。
- 响应体使用统一信封；失败时必须设置 `success: false` 和稳定的 `error.code`，前端只根据这两个字段处理业务错误，不得依赖 HTTP 状态码分支。
- Handler 输入校验、认证授权失败、Usecase 错误映射和 Recovery 捕获的未预期错误，均遵循该约定；不得向客户端泄露内部错误、panic 内容或堆栈。
- `/healthz` 与 `/readyz` 属于部署探针，不面向前端协议，保留标准 HTTP 状态码：存活为 `200`，未就绪为 `503`。

## 3. 数据与事务

- 任何跨多张业务表的一致性写入必须由 Usecase 通过 `shared.TxManager` 包裹。
- Repository 必须使用上下文中的事务连接，不能绕过 `database.DB(ctx, db)` 新建独立写入。
- 生产环境禁止使用 `AutoMigrate`。所有表结构变更必须提供版本化 migration、回滚策略和升级说明。
- 所有业务表应明确主键、唯一约束、必要索引、外键策略和软删除策略。
- 密码、刷新令牌、一次性验证令牌只保存不可逆 hash，绝不记录明文到数据库、日志或追踪属性。

## 4. 异步与可靠性

### 4.1 Outbox

- 所有会触发外部副作用的领域事件，必须在同一数据库事务中写入 `outbox_events`。
- 不得在事务提交前直接向 Kafka、邮件服务或其他外部系统发送业务事件。
- Outbox 发布采用至少一次投递语义；生产者可能重复投递，消费者必须幂等。
- 消息 key 必须稳定、可追踪，并在消费者侧作为幂等处理的依据。
- Outbox 发布成功但回写 `published_at` 失败时允许重发，不得将其误认为消息丢失。
- 若 `cron` 支持多实例部署，必须使用事件 claim、租约或 PostgreSQL `FOR UPDATE SKIP LOCKED`，避免多个实例同时发布同一事件。

### 4.2 消费与死信

- 消费处理必须先建立或获取消费记录，再执行副作用。
- 成功处理后才提交 offset；失败路径必须有明确的重试、DLQ 或人工处理策略。
- 重试次数、重试间隔、DLQ topic 和回放目标必须配置化。
- DLQ 回放必须生成新的消息标识，同时保留原始消息标识和回放上下文。
- 新增事件必须同时定义：事件名、payload schema、生产方、消费方、幂等键和失败策略。

## 5. 配置与安全

- 所有环境差异通过配置文件和环境变量表达；代码中不得硬编码密码、DSN、API key 或生产地址。
- 示例配置只使用无效的占位值；真实密钥只由部署环境注入。
- 配置加载只负责解析、环境变量覆盖和确定性默认值；每个已启用组件必须在装配时校验自己的关键配置，例如数据库 DSN、JWT 密钥、Kafka broker、Redis 地址、邮件配置和 cron 间隔。
- 就绪检查负责验证该进程实际依赖的外部服务可达性，不得将网络探测隐式混入配置解析。
- 生产 JWT 密钥必须由安全密钥管理系统或环境变量提供，且不得使用默认值。
- HTTP 服务必须显式配置超时和可信代理；管理接口必须经过认证与授权中间件。
- 验证码、注册、登录、刷新令牌、验证邮件与密码重置等公开安全接口必须按客户端 IP 使用 Redis 限流；命中限流或限流基础设施不可用时，均通过稳定的 `error.code` 返回给前端。
- 日志、指标和追踪不得包含密码、token、验证码、完整邮箱验证链接或敏感 payload。

## 6. 测试与质量门槛

合入 `main` 前至少必须通过：

```bash
go test ./...
go vet ./...
go build ./cmd/server ./cmd/worker ./cmd/cron
```

新增或修改功能必须按风险补充测试：

- 领域规则和 Usecase：单元测试，覆盖成功、输入非法、依赖失败和边界条件。
- Repository：PostgreSQL 集成测试，验证事务、约束、索引相关查询和并发语义。
- HTTP Handler：路由级测试，覆盖认证、授权、请求校验和错误码。
- Outbox、Kafka、Redis：覆盖重复投递、重复消费、失败重试、重启恢复和上下文追踪。
- 安全相关功能：覆盖 token 过期、撤销、权限变更、未验证用户和错误凭证。

测试不得依赖开发者本机的固定数据库、端口、账号或残留数据。集成测试应使用独立的可配置测试环境。

## 7. 可观测性与运行规范

- HTTP 请求、Outbox 发布、Kafka 消费和定时任务必须继承或创建 OpenTelemetry trace。
- 日志使用结构化字段，至少包含事件名、业务标识、消息标识和错误信息。
- 关键任务必须记录成功数、失败数、耗时、重试数和积压量。
- 所有后台循环必须响应 `context` 取消，并在进程退出时完成有限时间的优雅关闭。
- 对外依赖必须配置超时；无限重试、无上限 goroutine 和无界队列均不允许。
- `server`、`worker`、`cron` 必须暴露独立的存活与就绪探针；就绪检查只验证该进程实际依赖和后台循环状态，循环异常退出时进程必须退出并由部署系统重启。

## 8. 提交、评审与发布

- 一个提交只完成一个逻辑目的；通用改进与业务代码不得混在同一提交中。
- 提交信息使用清晰类型，例如 `feat`、`fix`、`refactor`、`test`、`docs`、`chore`。
- PR 必须说明：问题、方案、兼容性影响、配置或 migration 变更、测试证据和回滚方式。
- 破坏性配置、数据库、事件 schema 或 API 变更必须提供迁移方案，不得静默替换。
- 模板发布前更新 `README.md`、示例配置、部署文档和变更日志。
- 发布 tag 使用 `template/v<major>.<minor>.<patch>`；重大不兼容变更必须升级 major 版本。

## 9. 模板验收清单

每次发布模板版本前确认：

- 三个进程均可使用示例配置启动。
- PostgreSQL、Redis、Kafka 的本地启动方式明确且可验证。
- 数据库 migration 可从空库执行，并可升级上一版本数据。
- 注册、认证、授权、Outbox 发布、消息消费和死信路径具备自动化测试。
- 生产配置没有默认弱密钥、真实凭证或本地地址依赖。
- README 包含启动步骤、架构图、配置说明、测试命令和升级方式。
- 关键运行限制已写明：Outbox 语义、消费幂等要求、cron 部署模型和 DLQ 处理流程。
